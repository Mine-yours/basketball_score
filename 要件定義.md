# プロジェクト概要
バスケットボールの試合中にリアルタイムでスタッツを記録する、現場の入力スピードに特化したWebアプリケーション（SPA）のプロトタイプを作成してください。
入力負荷を極限限まで下げる「BasketPad2」のような神UI（自動判定、モーダル連動）を実装することが最大の目標です。

# 技術スタック
- React, TypeScript, Tailwind CSS, Lucide Reactアイコン
- 状態管理: React Hooks (useState, useMemo) または Zustand 等のイベント駆動型に適したもの

# 1. データアーキテクチャ（最重要：イベント駆動型）
数値を直接加算するのではなく、「イベントログ（Play-by-Play）の蓄積と、そこからの自動算出」という設計を徹底してください。

## 型定義の要件
- `Team`: 'home' | 'away'
- `ActionType`: '1P_MAKE' | '1P_MISS' | '2P_MAKE' | '2P_MISS' | '3P_MAKE' | '3P_MISS' | 'OR' | 'DR' | 'ASSIST' | 'STEAL' | 'BLOCK' | 'TO' | 'FOUL'
- `Player`: id, number, name, team, isOnCourt (boolean)
- `GameEvent`: id, timestamp(実時間), gameClock(試合の残り時間文字列), period(Q1等), team, playerId, action, x?(座標), y?(座標), assistPlayerId?(アシストした選手のID)

## データの永続化と算出
- `events` 配列が更新されるたびに `localStorage` に保存し、リロード時のデータ消失を防いでください。
- 画面上の「両チームの合計スコア」「チームファウル数」「各選手の個人得点(Pts)・ファウル数(F)」は、すべて `events` 配列から `useMemo` で自動算出（派生状態）してください。

# 2. UIレイアウト
画面はスクロール不要の1画面で完結させます。

- **ヘッダー（スコアボード）**: 両チームの合計得点、現在のクォーター、ゲームクロック（再生/停止ボタン付き）、チームファウル数。
- **サイド（選手リスト）**: 左右にHome/Awayの現在コートにいる5名を表示。各選手のボタン内に「現在の得点(Pts)とファウル数(F)」をバッジ等で小さく表示。各チームのリスト末尾には個人が特定できなかった時用の「チーム（背番号なし）」ボタンを配置。
- **センター（入力エリア）**: 上部にインタラクティブな「ハーフコート図（SVG）」、下部に「アクションパッド（FT, リバウンド, スティール, ブロック, ターンオーバー, ファウル, Undoボタン）」。

# 3. コア機能と操作フロー（神UI要件）

## ① シュート入力と2P/3P自動判定 ＆ アシスト連携フロー
コート図(SVG)と連動した、入力ステップを最小化するモーダルUIを実装してください。
1. **コートタップ**: 選手を選択した状態でコートのSVGをタップすると座標(x, y)を取得。
2. **2P/3Pの自動判定**: リングの座標を基準に、タップした座標が3Pラインの内側か外側かを数学的に判定してください。
3. **ステップ1モーダル（成否）**: タップ直後、「成功(Make)」と「ミス(Miss)」の2択ボタンを表示。点数（2Pか3Pか）はステップ2の自動判定に従います。ミスを選んだ場合はここでイベント保存し完了。
4. **ステップ2モーダル（アシスト）**: ステップ1で「成功」を選んだ場合のみ、連続して「アシストはありましたか？」という画面に切り替えます。選択肢は「アシストなし」および「コートにいる同チームの他4名（自分自身は除外）」とします。
5. **イベント生成**: 選択完了後、`{ action: '2P_MAKE', playerId: '打った人', assistPlayerId: 'アシストした人(いれば)', x, y }` のように、1つのイベントオブジェクトとして `events` に保存してください。

## ② リバウンドの自動判定（OR / DR）
画面上のボタンは「O.REB」「D.REB」を分けず、「リバウンド」という1つのボタンのみにしてください。
- **判定ロジック**: リバウンドボタンが押された際、`events` 配列の最新のイベントが「シュートミス (MISS)」かどうかを確認します。ミスの直後だった場合、[シュートを打ったチーム] と [リバウンドを取った選手のチーム] を比較し、同じなら `OR`、違うなら `DR` として自動記録してください。

## ③ 柔軟な入力順序（順不同受付）
「選手を選んでからコート(アクション)」でも、「コート(アクション)を選んでから選手」でも受け付けるように状態管理（pendingStateなど）を工夫してください。必要な情報が揃った瞬間にイベントを発火させます。

## ④ 選手交代（サブスティテューション）
初期モックデータとして、各チーム10名ずつ（isOnCourt: trueが5名、falseが5名）のデータを持たせてください。
- 画面端に「選手交代モード」のトグルボタンを設置し、コート上の選手とベンチメンバーを入れ替えられるUIを実装してください。

## ⑤ 確実なUndo（1つ戻る）
アクションパッドの「Undo」ボタンを押すと、`events` 配列の末尾の要素（最新のイベント）を1つ削除（pop）します。派生状態は自動的に再計算され、スコアも完全に1つ前の状態に戻るようにしてください（アシストデータも本体のシュートイベントに含まれるため同時に消えることになります）。

# 出力要件
バックエンドは不要です。モックデータを使用し、ブラウザ上で上記の複雑な状態遷移とUIコンポーネントが完全に動作するReactコード（単一ファイル、または明確に分割されたファイル群）を出力してください。
